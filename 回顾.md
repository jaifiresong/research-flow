# zhipin.com 风控研究总结

## 需求场景

通过 CDP（Chrome DevTools Protocol）自动化控制 BOSS直聘（zhipin.com），实现打开页面、获取可交互元素快照（snapshot）、根据快照结果操作页面（click/fill）的功能。目标是将浏览器自动化能力集成到 AI Agent 工作流中。

---

## 尝试的方案

### 方案一：agent-browser CLI (`--cdp` 模式)

使用 `agent-browser --cdp 9222` 连接到用户已有的 Chrome 实例（通过 `--remote-debugging-port=9222` 启动），执行 snapshot 等操作。

```
agent-browser --cdp 9222 open https://www.zhipin.com/web/geek/jobs
agent-browser --cdp 9222 snapshot -i
```

### 方案二：agent-browser daemon 模式 (`connect`)

先 `agent-browser connect 9222` 建立常驻连接，后续命令复用连接。

### 方案三：nodriver（Python CDP 库）

使用 `nodriver` Python 库连接已有 Chrome CDP 端口，执行导航和页面读取。

### 方案四：agent-browser + nodriver 混合

nodriver 负责导航（绕过风控），agent-browser `--cdp` 负责 snapshot（复用其 accessibility tree 和 ref 格式）。

### 方案五：原生 WebSocket + CDP

不依赖任何第三方自动化库，直接用 Python `websockets` 连接 Chrome CDP 端口，手动构造 JSON-RPC 命令。

## 遇到的问题

### 问题一：agent-browser --cdp 触发风控

`agent-browser --cdp 9222 snapshot -i` 执行后，页面在约 100ms-2s 内重定向到 `about:blank`（白屏）。

确认 `navigator.webdriver = true`，这是反爬检测的头号标志。

### 问题二：agent-browser daemon 模式与 nodriver 无法共享 tab

`agent-browser connect` 启动的 daemon 创建了自己的浏览上下文（新 tab），看不到 nodriver 导航的页面。

### 问题三：nodriver `browser.get()` 不稳定

偶尔返回"新标签页"（`chrome://newtab/`），导航失败。需改用 `main_tab` + 原生 CDP `Page.navigate`。

### 问题四：`/json/new` HTTP API 创建页面触发检测

通过 `PUT /json/new?url=` 创建新 tab 后导航到 zhipin，页面被重定向到 `about:blank`。

---

## 问题定位过程

### 1. 排除 `navigator.webdriver`

测试发现 nodriver 连接后 `navigator.webdriver = false`，agent-browser 连接后 `= true`。但这并非全部原因——后续测试中即使 `webdriver = false` 仍可能触发。

### 2. 查阅 GitHub Issues

发现 `vercel-labs/agent-browser` 相关 issue 和 `jackwener/OpenCLI#1757` 讨论了类似问题：

> CDP 的 `chrome.debugger.attach()` 可被页面 JS 观测到——站点可以 hook `window.chrome.debugger`，监听 `Runtime.enable` 事件暴露，或使用时间指纹检测。BOSS直聘在检测到 debugger attach 后在约 100ms 内重定向到 `about:blank`。

### 3. 阅读 agent-browser 源码定位可疑代码

`cli/src/native/browser.rs` 的 `enable_domains()` 方法在连接时发送以下 CDP 命令：

```rust
Page.enable
Runtime.enable
Runtime.runIfWaitingForDebugger  // 明确通知"调试器已附着"
Network.enable
Target.setAutoAttach             // 自动附着所有 iframe
```

其中 `Runtime.runIfWaitingForDebugger` 和 `Target.setAutoAttach` 为高度可检测的调试器信号。

### 4. 二分测试 CDP 命令组合

通过 nodriver 逐条发送 CDP 命令测试，发现：

- **单独发送** `Page.enable`、`Runtime.enable`、`Runtime.runIfWaitingForDebugger`、`Network.enable`、`Target.setAutoAttach`——均不触发风控
- **在导航前调用** `Page.enable` + `Runtime.enable` → 页面加载后 2-3 秒被重定向
- **不调用任何 `.enable()` 命令** → 完全不触发风控

### 5. 对照 nodriver 源码确认差异

分析 nodriver 的 `Tab.attach()` 源码发现：

```python
# nodriver 的 Tab.attach() 中
# 只调用 Target.attachToTarget 和可选的 Target.setAutoAttach
# 不调用 Page.enable 和 Runtime.enable
```

nodriver 的 `send()` 方法直接发送 CDP 命令（如 `Runtime.evaluate`、`Page.navigate`），**不依赖这些 domain 的 `.enable()` 事件订阅**。

### 6. 最终确认

在 Google 搜索获得关键信息：

> `Target.attachToTarget` with `flatten:true` is detected by boss. CDP's debugger attach is observable by page JS.

### 根因结论

**`Page.enable` 和 `Runtime.enable` 是 CDP 的事件订阅命令，它们本身不影响 `Page.navigate`、`Runtime.evaluate` 等方法的功能调用。但这些命令会在页面中创建可被检测的 CDP 会话状态。zhipin 的 JavaScript 检测到这些状态后，延时 2-3 秒重定向到 `about:blank`。**

## 最终解决方案

### 技术方案：原生 WebSocket + CDP，不发送 `.enable()` 命令

| 文件 | 行数 | 职责 |
|------|------|------|
| `cdp.py` | 59 | WebSocket 连接管理、JSON-RPC 消息收发 |
| `browser.py` | 167 | 高层 Browser API（open、snapshot、click、fill、evaluate） |

外部依赖：仅 `websockets`（Python 异步 WebSocket 库）。

### 核心设计

```python
# start(): 创建页面并连接
browser WS → Target.createTarget(about:blank) → 关闭 browser WS → 连接页面 WS

# open(url): 导航
Page.navigate(url) → await asyncio.sleep(wait)

# snapshot(): 获取交互元素
Accessibility.enable() → Accessibility.getFullAXTree() → 过滤角色 → 分配 @eN ref

# click(ref): 点击
DOM.resolveNode(backendNodeId) → Runtime.callFunctionOn → element.click()

# evaluate(js): 执行 JS
Runtime.evaluate(expression, returnByValue=True)
```

**关键原则：不调用 `Page.enable()`、`Runtime.enable()`、`Runtime.runIfWaitingForDebugger()`、`Target.setAutoAttach()`。这些命令仅用于事件订阅，对方法调用非必需。**

### 与 agent-browser 的差异

| 方面 | agent-browser | 本方案 |
|------|---------------|--------|
| enable_domains | Page/Runtime/Network enable + runIfWaitingForDebugger + setAutoAttach | 仅按需 `Accessibility.enable` |
| 连接方式 | `Target.attachToTarget(flatten=true)` + session ID | 直连页面 WebSocket URL |
| 页面创建 | `Target.createTarget` 或 daemon 管理 | `Target.createTarget`（一次性） |
| 依赖 | 11MB Rust 二进制 | websockets + stdlib |
| 风控触发 | 是（zhipin 检测 CDP session） | 否 |

### 验证结果

- 打开 zhipin 职位列表页：正常加载，81 个交互元素
- 3 秒后检查：页面未被重定向，稳定保持
- snapshot 完整输出：`@eN [role] "text" → url` 格式
- click 跳转职位详情：正常导航，详情页加载完成





根因一句话：Page.enable / Runtime.enable 只是 CDP 事件订阅，不影响功能调用。agent-browser 强制发了，zhipin 检测到 CDP session 状态后重定向到白屏。不调它们就绕过。

CDP 协议里有两类消息：
命令（Command）：你发请求，Chrome 返回结果。
→ Page.navigate({url: "..."})     ← {"frameId": "..."}
→ Runtime.evaluate({expression})  ← {"result": {"value": 123}}
这类不需要 enable，直接调就行。
事件（Event）：Chrome 主动推送通知。比如页面加载完成、控制台输出了东西、网络请求开始。
← Page.loadEventFired       ← Runtime.consoleAPICalled
这些事件必须先调用对应 domain 的 .enable() 开启订阅，Chrome 才会推送给你。.enable() 的作用就像订阅某个频道的通知——不开就收不到，但不影响你主动发命令。
所以 Page.enable() 是订阅"页面加载完成"等事件，Runtime.enable() 是订阅"控制台输出"等事件。不订阅它们，你照样可以调 Page.navigate、Runtime.evaluate、Accessibility.getFullAXTree——只是收不到事件通知而已。
agent-browser 为了监控页面状态（等待页面加载、监听 iframe 附着、网络请求追踪等），必须订阅这些事件。但 zhipin 检测到 Runtime.enable 这条订阅请求后，知道 CDP 调试器连上了，于是延时 2-3 秒把页面重定向到 about:blank。我们不订阅事件、只发命令，就绕过了这个检测。
